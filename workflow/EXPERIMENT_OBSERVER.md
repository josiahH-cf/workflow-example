# Experiment Observer

> **What this is:** An instruction file that gets copied into the test project (`hello-verify/`) and referenced by AGENTS.md. It makes the executing agent self-log every action, failure, retry, and decision to `experiment-log.md` in the project root.
>
> **Why it exists:** The human running the test should not be the one catching and recording errors, context poisoning, unclear instructions, or workflow breakdowns. The agent records them autonomously as structured, machine-parseable entries that a second agent can later analyze and convert into actionable repo changes.
>
> **Target file in test project:** `workflow/EXPERIMENT_OBSERVER.md`

---

## Contract

You are operating under experiment observation. In addition to your normal workflow duties, you MUST maintain `experiment-log.md` in the project root as an append-only log.

### When to Log

Log an entry after **every** one of these events:

1. **Phase transition** — entering or exiting a workflow phase
2. **File read** — reading any file as part of workflow execution (batch into one entry per phase)
3. **File write** — creating or modifying any file
4. **Command execution** — running any terminal command (build, test, lint, git)
5. **Decision** — making a choice between alternatives
6. **Failure** — any error, test failure, build failure, or blocked state
7. **Retry** — repeating an action after a failure
8. **Instruction issue** — encountering unclear, ambiguous, contradictory, or missing instructions
9. **Context confusion** — reading instructions that conflict with each other or with observed project state
10. **Skip** — skipping a file, step, or action (and why)
11. **Tool handoff** — transferring work to a different tool or session
12. **Bug track** — logging or fixing a bug via `/bug` or `/bugfix`

### Entry Format

Append each entry to `experiment-log.md` using this exact format:

```markdown
---

## [PHASE_NAME] — [ISO-8601 TIMESTAMP]

**Severity:** info | warning | error | critical
**Tool:** Copilot | Claude Code | Codex | [other]
**Action:** [one-line description of what was attempted]

**Files read:**
- `[path]` — [why it was read]

**Files written:**
- `[path]` — [what was created/changed]

**Commands run:**
- `[command]` → exit code [N]
  ```
  [first 5 lines of stdout/stderr, or "success — no output"]
  ```

**Decision:** [what was decided and why, or "N/A"]

**Failure:** [error text and root cause assessment, or "none"]

**Instruction clarity:**
- [clear | ambiguous | contradictory | missing] — [which file/section and what was unclear]

**Retry:** [yes — count: N, trigger: "..." | no]

**Files skipped:**
- `[path]` — [why it was skipped]

**Proposed workflow fix:**
- **File:** `[repo-relative path in workflow repo]`
- **Change:** [what should be added/removed/rewritten]
- **Why:** [evidence from this entry]

[or "none — this step worked as expected"]
```

### Severity Levels

| Level | When to use | Example |
|-------|-------------|---------|
| `info` | Normal successful operation | Phase transition, file created, tests pass |
| `warning` | Instruction unclear but workaround found | Ambiguous spec, missing example, used guess |
| `error` | Failure that blocks progress | Build error, test failure, missing file |
| `critical` | Workflow broken, cannot proceed | Contradictory instructions, infinite loop, state corruption |

### Rules

1. **Never skip logging.** Log even when everything succeeds — `info` entries establish the baseline.
2. **Never edit or delete previous entries.** The log is append-only. Corrections go in a new entry.
3. **Log context confusion immediately.** If two files say conflicting things, log both file paths, the conflicting statements, and which one you followed.
4. **Log instruction gaps.** If you had to guess or infer something not written in the workflow docs, log what you guessed and what doc should have told you.
5. **Log every retry.** Include what failed, what you tried differently, and whether it worked.
6. **Be specific about file paths.** Always use repo-relative paths from the workflow source repo (e.g., `template/workflow/PLAYBOOK.md`) for proposed fixes, not paths in the test project.
7. **One entry per event.** Don't batch multiple unrelated events. One phase transition = one entry. One build failure = one entry.
8. **Proposed fixes are optional for `info` entries** but required for `warning`, `error`, and `critical` entries.

### Initial Entry

When you first read this file, create `experiment-log.md` in the project root (if it doesn't exist) and write this header:

```markdown
# Experiment Log

> Auto-generated by the executing agent per `workflow/EXPERIMENT_OBSERVER.md`.
> Append-only. Do not edit previous entries.

**Test project:** hello-verify (Windows Hello World — C/mingw-w64)
**Started:** [ISO-8601 timestamp]
**Workflow repo version:** [git commit hash or "unknown"]
**Tool:** [Copilot | Claude Code | Codex]

---
```

Then log your first entry: reading this observer file and the initial scaffold state.

### How This Gets Analyzed

After the test run completes, `experiment-log.md` is copied to the workflow repo at `docs/verification/experiment-log.md`. A second agent then:

1. Parses all entries
2. Filters for `warning`, `error`, `critical` severity
3. Groups proposed fixes by target file
4. Generates entries in `docs/verification/agentic_test_change_proposals.md`
5. Produces a summary score in `docs/verification/agentic_test_log_template.md`

This closes the loop: run workflow → agent logs issues → second agent proposes fixes → human reviews and merges.
